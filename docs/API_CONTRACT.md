# The API Contract (Team 1 🤝 Team 2)

Because Team 1 is preparing the databases offline and Team 2 is querying them online, both teams **MUST** agree on the following 4 rules before writing any Python code. If these formats don't match exactly, Team 2's AI agents will write search algorithms that retrieve empty results.

---

## 1. The Core ID (Primary Key)
Every single keyframe extracted from a video must have a universal, unique identifier. Both Turbovec and Elasticsearch must use this exact same ID format. This is critical because Team 2 will need to combine visual scores from Turbovec with text scores from Elasticsearch (using SRRF), and they need the ID to match them up.

*   **Format Agreement:** `{video_id}_{frame_index}`
*   **Example:** `L01_V001_0145` (Meaning: Video L01_V001, Frame 145)

## 2. Elasticsearch Document Schema
When Team 1 inserts ASR/OCR text into Elasticsearch, they must use the following JSON schema. Team 2 will write their BM25 search algorithms expecting these exact field names:

```json
{
  "frame_id": "L01_V001_0145",
  "video_id": "L01_V001",
  "timestamp_seconds": 14.5,
  "ocr_text": "Trường Đại học Bách Khoa", 
  "asr_text": "Xin chào các bạn sinh viên"
}
```

## 3. Turbovec Storage Agreement
Turbovec stores the mathematical embeddings generated by SigLIP and BEiT-3. Team 2 needs to know *which* image the vector belongs to. 
*   **Agreement:** Team 1 must attach the `frame_id` alongside the vector when inserting it into Turbovec.

## 4. File System Paths (For the UI)
When Team 2 retrieves a `frame_id` from the database, their Streamlit Web UI needs to display the actual image to the user. This means Team 1 must save all extracted keyframes in a highly predictable folder structure so Team 2 can just construct the file path dynamically.

*   **Path Agreement:** `data/keyframes/{video_id}/{frame_id}.jpg`
*   **Example:** `data/keyframes/L01_V001/L01_V001_0145.jpg`
